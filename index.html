Intake Portal

Project Overview

The purpose of the project was to implement an intake portal for a client’s application process across two streams. The existing process was carried out manually. All forms were filled by hand and records were entered manually into Excel/CTMS database. As a result, this process was inefficient, time-consuming, and prone to human error. Thus, there was a need to automate this process.

The Solution

The intake portal enables stakeholders to receive applicant data online, review the data, request more information, add additional information to client profiles, store/archive client profiles, approve and deny applications, and push data to the CTMS database upon applicant acceptance.

My Role

UX Designer and Front-End Developer

Tools

  • Sketching, brainstorming - Pen, paper, Whiteboard, markers
  • Mockups/Wireframes - Balsamiq Mockups and MockupsToGo
  • User Interface Design - Sketch
  • Prototyping - Invision
  • Development - written in HTML, CSS, Velocity, AngularJS

UX Approach

Stakeholder Interviews

In order to properly define the project scope and requirements, I conducted a series of interviews with the project owners. The goals of this meeting were as follows:

  • Gain an understanding of the current business process
  • Identify pain points
  • Define users and user groups
  • Define specific process/workflow for each user/user group
  • Define project vision, success metrics and potential hurdles

Defining Project Requirements

The following requirements were determined as a result of the interviews:

  • Intake Portal for application and selection process up until approval
  • Intake portal to support both Homeward bound and Boundless possibilities for women for each cohort
  • User-friendly, easy to use, and mobile responsive
  • Email notification at specific milestones in application process
  • Granular permissions with restricted access
  • Data push to CTMS servers
  • CTMS API to integrate with Portal data (intake information)

Redesigning Membership Directory Feature

Project Overview

The goal of this project is to improve Readyportal's Membership Directory feature by making it more user friendly and modern.

The Current Situation

Below is a the current default membership directory implementation. Other than a few style changes (to match client implementation), this feature has never been upgraded.

The Solution

The proposed solution makes it easier for admin users to add new users, edit and delete current users, edit user profile, and search within the Membership List view by reducing the number of steps taken to achieve these actions.

Project Duration

This project duration was a 48 hour sprint in between projects

My Role

UX Designer and Front-end Developer

Tools

  • Brainstorming - Whiteboard, Pen and paper
  • User Feedback - Google Forms (for questionnaire)
  • Wireframing - Balsamiq Mockups
  • Prototyping/Devlopment - HTML, CSS, Velocity, Javascript

UX Approach

Identifying the Problem

In order to identify our users' pain-points, I conducted interviews with five power users. The questions asked are as follows:

  1. Do you use the membership directory feature?
  2. If yes, how often do you use this feature per week?
  3. What is your objective when you use this feature?
  4. Can you show me how you use this feature?
  5. What do you like about this feature?
  6. What frustrates you about this feature?
  7. Do you have any suggestions to improve this feature?

Recommendations

The recommendations from the above exercise were:

  •  Reduce the number of steps it takes to edit user information
  •  Simplify search function
  •  Feature should be responsive/look just as good on mobile
  •  Improve look/feel of membership directory feature – make it more modern
  •  Improve pagination
  • Make it easier to add/remove users from user groups

Wireframes

With the user interview feedback in mind, I designed the following wireframes in Balsamiq. The wireframes includes the following changes:

Search functionality in the list view:

The current implementation only had an advanced search functionality. Thus, users had to click "Advanced Member Search",  enter a search query,  and click submit for each search. The proposed solution has a live search in the list view, resulting in only one action.

Add new user button in the list view:

In the current implementation, new users could only be added from the "Admin Wizard" screen which is separate from the Membership Directory screen. This was added in order to make it easier for users to manage all user information from the same view.

Edit button in the list view:

The process for editing user information in the current implementation was as follows:

  • Click user name to view user profile
  • In the user profile view, click edit user information

With the edit button in the list view, users are able to skip this process.

Adding/Removing Users from group using a checkbox:

In the current implementation, this process was as follows:

  • Click other actions
  • Select Add to Group or Remove from group
  • User is directed to Group Security page
  • User clicks submit from Group Security page

The proposed solution completes these actions in just two clicks., as follows:

  • Click other actions
  • Check or uncheck to add or remove from group

Working Solution

Once the wireframes were tested and approved, I went straight to code this time. We decided to use the default Material Design template due to time constraints. Below are live screenshots of the feature:

Constraints

  • Time: I had 48 hours to complete a redesign and development.
  • Less than ideal user sample size: I was only able to interview 5 users in such a short time frame. Ideally, a larger sample size would produce better results.
  • Design: If I had more time, I wouldn't use a default Material Design template, because the rest of our product (and custom implementations) do not follow the same theme. As a result, the Membership Directory page is not cohesive with other parts of our product.

Conclusions

It's amazing what can be achieved in such a short period of time when we let go of our need for perfection. The challenge here was to produce a minimum viable product in a short time frame, without compromising in terms of providing value.

In Car GPS Navigation

Project Overview

The purpose of this project was to apply user-centered design methods to improve the functionality of the GPS Navigation System.

My Role

UX Designer

Project Duration

This project was completed in a 6 week time-period in 2014.

Tools

Balsamiq, Adobe Photoshop

UX Approach

For this project, the requirements were gathered using the following processes:

User Personas

Task Analysis

Task analysis in this section has been carried out on a generic navigational display in a Toyota Rav4.

Getting directions to an unknown address using in-car GPS navigation system

  1. Turn the car on
  2. Press “Dest” button
  3. Press “Address”
  4. Select “Street Address”
  5. Enter house number using touchscreen
  6. Enter street name using touchscreen
  7. Select correct address from Match list
  8. Push “Go to” button
  9. To choose alternate routes, click “x routes” button
  10. Select desired route
  11. Click ok

Getting directions to a previous destination using in-car GPS navigation system

  1. Turn the car on
  2. Press “Dest” button
  3. Press “Previous Destinations”
  4. Select destination from list
  5. Push “Go to” button
  6. To choose alternate routes, click “x routes” button
  7. Select desired route
  8. Click ok

Developing Requirements

From the above tasks, the following requirements were derived:

Developing Requirements

Touch Sensitive User Interface:

The system needs to display GPS information to the user and also allow the user to input addresses. An easy way to do this is using a touchscreen display because it serves both input and output purposes.

Easy to Use:

ince majority of the users will be driving and using the system simultaneously, it is necessary for tasks to be completed in as few steps as possible.

Input address information:

The GPS system needs to be able to process address data entered by the user.

Save previous destinations:

The GPS system should save previously entered locations, so that the user does not always have to enter addresses every time the system is used.

Ability to select desired route:

The system should give the user the option to select from a variety of routes to the location.

User Flow

The user flow for the system is as follows:

  1. User selects either new address or previous destination option
  2. If previous destination selected, go to step 4. User enters address
  3. User selects matching address from list. Go to step 5
  4. User selects destination from list of previous destinations
  5. User selects desired travel route

Design Review

The design review was carried out using the System Usability Scale, SUS. The SUS provides a quick and effective method to measure the usability of a system. It is easy to administrate, and can be used in small sample sizes.

Three types of participants were examined:

  1. Novice (N): This is someone who has little to no experience using secondary car displays. This encompasses people without cars, those who depend on public transit and/or those who drive older cars without navigational displays.
  2. Stakeholder (S): This is someone who uses a navigational display system on a daily basis. This user typically drives a newer model car with a secondary display.
  3. Casual user (CU): An occasional user.

Advantages and Limitations

The biggest strength of the GPS display design is the ease of use, as suggested by the high scores of item 3 and item 8. In answering these two questions, it was realized that the users usually find the system easy and not cumbersome to use. Additionally, the test users stated that they do not need to learn a lot of content before using the system. The lowest score being item number 5 and 6, exposes the major weakness in this design being inconsistencies with the primary display design; however, future improvements can be made to make the styling more consistent.

Design Justifications

Navigation

The secondary display is a touch screen device. Taking this into consideration, the aim was to design an interface that reduced the number of interactions between the user and the interface. As a result, tabbed navigation was implemented to switch between the three navigation tasks. This greatly reduced the interactions, when compared with the current navigation systems used in cars. The selected tab is highlighted; thus preventing the user from getting lost as they are always aware of the selection at every point in time. Within the selected tabs, buttons were implemented for navigation. Back buttons were used to allow the user freely navigate back and forth within the system.

Layout

he in-car GPS systems are typically small in size. As a result, it was necessary to use the whole screen for visibility purposes. The tabs/headers were placed at the top of the screen. The lists and dialogue boxes were centered, and the buttons were justified to be inline with the lists/dialogue boxes when used.

Colour & Icon Design

The colours and icons were kept very simple and minimal, using black, white, greys and green. The selected tabs all have a dark grey background colour. Non-selected headings and buttons are lighter grey, while selected buttons are green. The text colour varied from white against a grey background, black against a white background, and green when selected in an input box. The use of green for selected items gives the users a visual feedback, making them aware of what is happening.

Fonts & Typography

Uppercase typography was used across the system to enhance visibility. The use of the system would simultaneously be driving; thus, it is necessary to have the fonts as easy to read as possible in order to save time and enhance efficiency. The headers utilized larger sized fonts in order to enhance salience. Also, the same font (bold Montserrat)  was used across the system for consistency purposes.

Controls

The in-car GPS system uses a touch-screen interface, and the accepted gesture is a push/press action. The at rest buttons are grey, while the active buttons are green. Scrollbars are also utilized for list navigation.
